FOH ↔ HF Integration Strategy

System Connection Design  ·  Human First Foundation  ·  February 2026  ·  CONFIDENTIAL — DRAFT

Executive Summary

Recommendation: One seam. Context travels once. VAPI is invisible to FOH.
Situation

Two systems are being built in parallel: a web-facing Front of House (FOH) that captures caller identity and community interest, and HF — an AI voice brain that serves personalised prompts and processes call transcripts. Both must work together to deliver a coherent caller experience.

Complication

The natural instinct is to integrate deeply — FOH injecting context at call time, systems calling each other in both directions, VAPI events fanning out to both. This creates coupling that breaks autonomy: latency in the call path, cascading failures, shared secrets, and two teams blocking each other on every change.

Question

How do we connect two independent systems so that a caller captured by FOH receives a fully personalised HF voice session — without either system owning the other?

Answer

Connect them at a single point, before the call. FOH registers the caller in HF with a rich context payload. HF primes the prompt. When the call happens, VAPI talks only to HF. FOH steps aside entirely. Post-call, HF pushes a summary webhook back to FOH. Two API calls. No shared infrastructure. No coupling in the live call path.

Three supporting principles
1

Pre-call is the only integration moment. All FOH→HF data transfer happens at registration, not at call time. The live call path is VAPI↔HF only.

2

Context seeds memory, not the prompt. FOH capture data (community, interests, onboarding answers) enters HF as memory entries, flowing naturally into prompt composition. No custom prompt injection needed.

3

Secrets are asymmetric and minimal. HF issues FOH one API key. FOH issues HF one webhook secret. VAPI's secrets stay between VAPI and HF. No handshake, no shared auth layer.

Supporting Detail

1. Pre-call is the only integration moment

The problem with in-call integration. VAPI imposes a hard 7.5-second deadline on the assistant-request event. Any synchronous FOH→HF call at that moment — to inject context, look up session state, verify enrollment — consumes that budget. One slow FOH response and the caller hears silence or gets a fallback prompt.

The topology that works separates the integration into three clean phases:

REGISTRATION  (FOH → HF, once, async-tolerant)
──────────────────────────────────────────────────────────
FOH  POST /api/callers/register
       { name, phone, community, context }
HF   → creates Caller → enrolls in course → seeds memory → composes prompt
HF   → 200 { callerId, promptReady: true }

CALL TIME  (VAPI → HF only, 7.5s budget, FOH invisible)
──────────────────────────────────────────────────────────
VAPI  POST /api/vapi/assistant-request  →  HF returns pre-built prompt
VAPI  POST /api/vapi/webhook            →  HF ingests transcript + runs pipeline

POST-CALL  (HF → FOH, async push)
──────────────────────────────────────────────────────────
HF   POST {foh-callback-url}
       { callerId, summary, topicsHit, nextSession }

FOH is active exactly once in the call lifecycle: at registration. Everything else runs without it. Critically, failure modes are contained: if FOH goes down, calls still work because prompts already exist in HF. If HF goes down, FOH still captures callers and can queue registrations for retry.

2. Context seeds memory, not the prompt

The naive approach — FOH sends a custom system prompt string to HF at registration — creates a maintenance problem. Every time HF's prompt template evolves, FOH breaks. The two teams become coupled at the content layer, meaning a change to pedagogy or persona requires a co-ordinated release across both systems.

The correct approach: FOH sends structured facts. HF decides how to use them.

POST /api/callers/register { "name": "Jane Smith", "phone": "+447700900123", "communityId": "abc123", "context": { "interest": "Improve sales pitch confidence", "experience": "3 years B2B, struggling with objection handling", "source": "website", "onboarding": { "goal": "close more deals", "preferred_style": "direct feedback", "availability": "weekday evenings" } } }

HF writes each context field as a CallerMemory entry. When the prompt is composed, these memories flow in through the existing pipeline — the same mechanism every other memory uses. Jane's first call feels like the AI already knows her, because it does. Her goals, her experience, her preferred style are all in the session prompt before she says a word.

FOH never needs to understand HF's prompt structure. It sends facts; HF decides what to do with them. This is the correct separation of concerns.

3. Secrets are asymmetric and minimal

Three directional relationships exist in this system. Each has exactly one issuer and one verifier. No secret is shared between more than two parties.

          FOH                      HF                     VAPI
          ───                      ──                     ────

          HF_API_KEY     ────────► (HF verifies)
                                                           │
                                   VAPI_WEBHOOK_SECRET ◄──┘ (FOH sets in VAPI dashboard)
                                   (HF verifies)

          CALLBACK_SECRET ◄─────── (HF signs outbound webhooks)
          (FOH verifies)
Secret Issued by Held by Verified by MVP?
HF_API_KEY HF FOH server HF Yes
VAPI_WEBHOOK_SECRET FOH/VAPI VAPI + HF env HF Yes (already live)
CALLBACK_SECRET FOH FOH + HF env FOH No — FOH polls for MVP

The HF_API_KEY lives on FOH's server only — never in the browser. VAPI's secrets are entirely between VAPI and HF; FOH colleagues set them in their VAPI dashboard but never use them in their own codebase. If either system is compromised, the blast radius is contained to that directional relationship only.

Decision Record

Decision Chosen Rejected Reason
Integration point Pre-call registration only In-call proxy / middleware 7.5s VAPI deadline; latency risk unacceptable in live call path
Context delivery Memory seed (structured facts) Custom prompt string injection Decouples content from prompt structure; survives prompt template changes
VAPI ownership FOH owns VAPI entirely HF owns VAPI / shared ownership FOH already built around it; clean single-owner model reduces conflict
Post-call delivery HF webhook push to FOH FOH polls HF Push enables real-time FOH UX updates; polling is acceptable for MVP
Auth pattern Asymmetric API keys per direction Mutual handshake / OAuth / shared secret Simpler, no shared state, contained blast radius per relationship
Unknown callers HF serves generic prompt; auto-creates bare record Block call if not pre-registered Graceful degradation; walk-in callers still get a session

Open Questions

  • Voice recordings. VAPI posts recordingUrl in the end-of-call report. Does FOH want this URL, or does HF store the audio and own it? This needs a decision before audio analysis work begins — time-limited URLs expire and the audio must be fetched promptly.
  • Unknown callers. If someone calls without pre-registration, HF serves a generic prompt and auto-creates a bare caller record (phone only, no community, no memory). Does FOH get notified? Should HF push a "new unregistered caller appeared" event so FOH can retrospectively capture them?
  • Community mismatch. A caller registered in Community A calls after moving to Community B. Who resolves this — FOH (re-registers with new community) or HF (re-enrolls on the fly)? Needs a clear owner to avoid silent mismatches in prompt content.
  • Prompt readiness SLA. Prompt composition after registration takes ~10–30 seconds (async AI pipeline). How long after registration does FOH guarantee it will wait before initiating a call? FOH should either poll for promptReady: true or respect a minimum delay before allowing a call to start.
Next action: Agree on the POST /api/callers/register contract — field names, community identifier format, context schema, and the promptReady polling mechanism. This is the entire interface between the two teams. Everything else can be built independently once this contract is signed off.
